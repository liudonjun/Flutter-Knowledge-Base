# 敏捷与迭代开发

> Flutter 团队的 Sprint 规划、Code Review、CI 与灰度发布实践。

## 迭代节奏

典型 **2 周 Sprint**：

| 阶段 | 活动 |
| --- | --- |
| 计划 | 从 backlog 选 story，估点（Story Point） |
| 开发 | 分支开发 + 每日站会 |
| 评审 | Demo 可运行 build（TestFlight / 内测 APK） |
| 回顾 | 改流程，不甩锅 |

Story 粒度：一个 Sprint 内可完成、可演示。

## 分支策略

```
main（可发布）
  └── develop
        └── feature/login-oauth
        └── fix/cart-quantity
```

- `feature/*` → PR 进 `develop`
- 发布周：`develop` → `release/x.y` → 合并 `main` 打 tag
- 热修复：`hotfix/*` 从 `main` 拉出

Flutter 项目 PR 应包含：`flutter analyze`、测试通过、截图/录屏（UI 变更）。

## Code Review 要点

| 维度 | 检查 |
| --- | --- |
| 架构 | 是否破坏 feature 边界 — [[17-目录结构设计]] |
| 状态 | 是否最小重建 — [[04-Riverpod详解]] |
| 性能 | 列表 builder、图片尺寸 |
| 测试 | 核心逻辑是否有单测 — [[15-mockito详解]] |
| 安全 | Token、密钥不进仓库 |

## CI 集成

```yaml
# 示例步骤
- flutter pub get
- flutter analyze
- flutter test
- flutter build apk --debug   # 或 build appbundle  nightly
```

- PR 必过 analyze + test
- main 分支 artifact 存内测分发（Firebase App Distribution /蒲公英）
- 符号文件存档 — [[14-应用打包与签名]]

## 定义完成（DoD）

- [ ] 验收标准满足
- [ ] 无已知 P0/P1 bug
- [ ] 文档/注释（公共 API）
- [ ] 已测 iOS + Android 至少一台真机
- [ ] 埋点/开关（若需要）已配置

## 灰度与发布

1. **内测** → TestFlight / 封闭测试轨道
2. **小流量** → 后端 feature flag 或按用户 ID 百分比
3. **全量** → 商店发布，监控 crash 率 — [[02-调试技巧详解]]

Flutter 侧配合 Remote Config / 自建开关，避免强制发版修文案。

## 与产品协作

- Sprint 中期同步：scope 裁剪优先于延期
- 设计稿变更走 ticket，避免口头改需求
- 技术债登记 backlog，每 Sprint 留 10–20% 容量
