# 架构设计面试题

> Flutter 项目分层、状态管理选型、依赖注入与可测试架构的高频题。

## 分层与职责边界

**题：你如何划分 Flutter 项目层次？**

**答**（常见 clean / feature-first）：

```
presentation  → UI、Controller/Bloc、页面状态
domain        → Entity、UseCase、Repository 接口
data          → API、DTO、Repository 实现、本地库
```

规则：**依赖向内**；domain 不 import `flutter/material.dart`。

**题：feature-first 和 layer-first 怎么选？**

| 结构 | 适用 |
| --- | --- |
| `features/auth/` 内分 data/domain/ui | 中大型、多业务线 |
| 顶层 `ui/` `data/` | 小 App、原型 |

见 [[17-目录结构设计]]、[[01-项目架构设计]]。

## 状态管理选型

**题：Provider / Riverpod / Bloc / GetX 如何选？**

| 方案 | 特点 | 适用 |
| --- | --- | --- |
| Provider | 官方生态、简单 | 中小项目 |
| Riverpod | 编译安全、可测试 | 推荐新项目 |
| Bloc | 事件驱动、规范强 | 复杂流程、大团队 |
| GetX | 快速、路由+状态一体 | 小团队需约定纪律 |

**答**：无银弹；说清 **全局态 vs 页面态**、是否需要 **事件溯源**、团队熟悉度。

**题：什么时候不需要全局状态管理？**

纯 `StatefulWidget` + 局部 `ValueNotifier` 即可；避免过度工程。

## 依赖注入

**题：get_it 和 Riverpod 能否一起用？**

**答**：可以。Service/Repository 用 [[10-get_it详解]] / [[21-injectable详解]]；UI 可监听状态用 Riverpod override 注入 Mock — [[46-provider_di详解]]。

**题：如何方便单测？**

- 面向接口编程
- 构造函数注入 Repository
- `getIt.registerLazySingleton<Api>(() => MockApi())`

## 可测试架构

**题：如何测 Bloc？**

[[29-bloc_test详解]]：`blocTest` 断言 state 序列。

**题：如何测 Widget？**

`ProviderScope`/`MaterialApp` 包裹；mock 网络用 `http.MockClient` 或 repository fake。

**题：集成测试边界？**

关键路径少量 E2E（登录、下单）；其余单元+Widget 测试覆盖。

## 模块通信

**题：跨 feature 跳转怎么设计？**

- 路由表集中 — go_router
- 不传 giant model；传 id，目标页自行拉数据
- 避免 feature A import feature B 的 UI

**题：Deep Link 与模块化？**

路由 path 稳定；模块只注册 route，不在 core 写业务 UI。
