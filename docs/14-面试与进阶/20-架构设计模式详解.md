# 架构设计模式

> Clean Architecture、分层与 Repository 在 Flutter 项目中的落地方式。

## 分层边界

常见三层（可简化为两层小项目）：

```
Presentation（Widget / Controller / Bloc）
        ↓
Domain（Entity、UseCase、Repository 接口）
        ↓
Data（API、DTO、Local DB、Repository 实现）
```

**原则**：UI 不直接调 Dio；Domain 不依赖 Flutter SDK。

---

## Repository 模式

- 接口在 Domain，实现在 Data。
- 统一缓存、错误映射、重试。
- 见 [[12-设计模式题]] 代码示例。

---

## UseCase

封装单一业务动作，便于组合与单测：

```dart
class LoginUseCase {
  LoginUseCase(this._authRepo);
  final AuthRepository _authRepo;

  Future<User> call(String email, String password) =>
      _authRepo.login(email, password);
}
```

小项目可用 Service 代替 UseCase，避免过度抽象。

---

## 模块与包拆分

| 规模 | 建议 |
| --- | --- |
| 单 app | `features/` 按功能垂直切分 |
| 多 app / 大团队 | Melos monorepo：`core`、`design_system`、`feature_*` |

依赖方向：`feature → core`，feature 之间不互相 import。
