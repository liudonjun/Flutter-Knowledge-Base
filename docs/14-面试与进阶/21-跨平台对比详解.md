# 跨平台对比详解

> Flutter vs React Native / 原生 / 小程序 的选型讨论框架（面试与立项均可使用）。

## 性能与包体

| 方案 | 性能 | 包体 | 说明 |
| --- | --- | --- | --- |
| **Flutter** | 自绘，接近原生 | 引擎体积 ~数 MB | 120fps 潜力，依赖 Skia/Impeller |
| **React Native** | JS Bridge，依赖原生组件 | 相对较小 | 新架构 Fabric 改善性能 |
| **原生双端** | 最优 | 单端无跨端引擎 | 两套代码成本 |
| **小程序** | 受限 | 轻 | 生态封闭，能力边界 |

面试：不贬低竞品，按**业务场景**选——重 UI 一致性、多端 → Flutter；强依赖原生 UI 策略 → 原生或 RN。

---

## 团队技能栈

- 已有 **Dart/Flutter** 团队 → 优先 Flutter。
- 大量 **React** 前端 → RN 学习曲线低。
- **原生 Android/iOS** 深厚 → 新模块可 Add-to-App 嵌入 Flutter。

[[技术栈拓展]]：全栈能力有助于架构讨论，但不是选型唯一因素。

---

## 生态与交付

- **Flutter**：pub.dev、官方维护路由/状态方案多；桌面/Web 同一套代码。
- **RN**：npm 生态；依赖原生模块质量参差。
- **交付**：Flutter 单测/Widget 测试成熟；CI 用 `flutter test` + `integration_test`。
