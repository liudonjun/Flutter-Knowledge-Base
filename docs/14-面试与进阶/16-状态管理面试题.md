# 状态管理面试题

> Provider、Riverpod、BLoC、GetX 等方案的对比与选型，附常见追问答题框架。

## 状态分类

| 类型 | 示例 | 典型方案 |
| --- | --- | --- |
| **局部 UI 状态** | 开关、输入框、Tab 索引 | `setState`、局部 `ValueNotifier` |
| **页面/模块状态** | 列表加载、表单 | Provider、Riverpod、Controller |
| **全局/跨页状态** | 登录用户、主题、购物车 | Riverpod、BLoC、GetX |
| **异步状态** | 网络请求、分页 | `FutureProvider`、`AsyncValue`、BLoC `AsyncSnapshot` |

**答法**：先区分状态作用域与生命周期，再选方案；避免「全局 Provider 管一切」。

---

## InheritedWidget 原理

**问：Provider 为什么能向下传递状态？**

- 底层是 `InheritedWidget`，在 Element 树向下查找最近祖先。
- `dependOnInheritedWidgetOfExactType` 建立依赖，数据变化时标记依赖 Element 重建。
- `ChangeNotifier` + `ListenableBuilder` / `Consumer` 在 notify 时触发 rebuild。

**追问**：与 `setState` 区别？——`setState` 只重建当前 State 子树；Inherited 可跨层级共享且按需重建。

## 方案对比

| 方案 | 优点 | 缺点 | 适用 |
| --- | --- | --- | --- |
| **setState** | 零依赖、简单 | 难共享、易滥用重建 | 单 Widget 内 |
| **Provider** | 官方推荐、易学 | 运行时错误、嵌套深 | 中小型项目 |
| **Riverpod** | 编译期安全、无 context | 概念多 | 中大型、新项目 |
| **BLoC** | 可测试、事件驱动清晰 | 样板代码多 | 企业级、复杂流 |
| **GetX** | 路由+状态一体 | 魔法多、架构约束弱 | 快速原型 |

**Riverpod vs Provider**：编译时检查、可在外部读 Provider、更好的 override 与测试。

**BLoC 优缺点**：单向数据流、易单测；Event/State 类多，小项目过重。

---

## 测试与可维护性

1. **可测性**：业务逻辑放 Notifier/Cubit/Repository，Widget 只负责渲染。
2. **最小重建**：`select`、`Consumer` 细粒度监听，避免整页 rebuild。
3. **依赖注入**：Repository 注入 Notifier，Mock 后测状态流转。
4. **反模式**：在 `build` 里 `Provider.of` 触发副作用；全局 Mutable 单例。
