# 系统设计面试题

> 移动端 Feed、缓存、推送、IM 与多端同步的系统设计面试题。

## 答题框架

1. **澄清需求**：DAU、离线要求、一致性、延迟
2. **估算**：QPS、存储、推送量（数量级即可）
3. **架构图**：客户端 / API / DB / 缓存 / 队列
4. **深入一点**：瓶颈、扩展、失败模式
5. **Flutter 侧**：状态、本地库、与原生分工

## Feed 流设计

**题：设计微博/朋友圈 Feed**

**答**：

```
客户端：分页 cursor + 本地缓存（Hive/SQLite）
API：GET /feed?cursor=xxx&limit=20
后端：推拉结合 — 写扩散（粉丝少）/ 读扩散（大 V）
```

Flutter：

- `ListView.builder` + 下拉刷新 / 上拉加载
- 图片 CDN + `cacheWidth`
- 点赞乐观 UI，失败回滚

参考 [[19-新闻应用实战]]、[[22-社交应用实战]]。

**追问**：热点 key？→ 缓存 + 限流；写合并。

## 离线缓存策略

**题：新闻 App 离线阅读**

| 层 | 策略 |
| --- | --- |
| 内存 | LRU 当前页图片 |
| 磁盘 | 文章 JSON + 缩略图路径 |
| 网络 | stale-while-revalidate：先展示缓存再后台刷新 |

**冲突**：以服务端 `updated_at` 为准；本地版本号字段。

Flutter：`dio` + interceptor 读缓存；或 Repository 封装 — [[02-数据存储基础]]。

## 推送与 Deep Link

**题：聊天 App 离线消息如何到达？**

见 [[13-消息系统设计]]：

- 在线：WebSocket — [[41-socket_io_client详解]]
- 离线：FCM data payload → 系统通知
- 点击：`conversation_id` Deep Link 进聊天页
- **正文敏感**：通知仅「您有一条新消息」，详情进 App 拉取

**题：如何保证不丢消息？**

服务端 seq + 客户端 `since_seq` 增量拉；推送只做 **唤醒**。

## 多端同步

**题：购物车多端同步**

1. 登录后 `GET /cart` merge 本地草稿 — [[21-电商应用实战]]
2. 改操作 `PATCH /cart` + 版本号 / etag
3. 冲突：服务端 LWW 或提示用户选

**题：已读状态同步**

会话级 `last_read_seq`；多端 WS 广播 `read_receipt`。

## 电商下单（简版）

**题：秒杀/下单 peak**

- 客户端：按钮 debounce、排队 UI
- 服务端：库存 Redis 预扣、消息队列异步落库
- 支付：[[11-支付系统设计]] 异步 notify

Flutter 只调 `create order` + 轮询状态，不参与库存逻辑。
