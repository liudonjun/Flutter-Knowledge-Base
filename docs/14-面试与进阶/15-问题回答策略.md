# 问题回答策略

> 结构化表达、边界澄清与「不会」时的应对方式。

## 先结论后展开

```
结论（1 句）→ 原理/理由（2～3 点）→ 项目例子（1 个）→ 可选延伸
```

示例：「Flutter 默认单线程，因为 UI 需要在同一 isolate 保证一致性；I/O 用 async，CPU 密集用 Isolate.run。」

避免：从 Dart 历史讲起；5 分钟仍无结论。

---

## 澄清需求

编码题与系统设计题先确认：

- 输入输出、数据规模、是否可持久化
- 是否需要考虑离线、权限、错误重试

一句话复述：「我理解是要实现 X，约束是 Y，我先给 MVP 方案可以吗？」

---

## 诚实与延伸

| 情况 | 做法 |
| --- | --- |
| **完全不会** | 承认 + 相关已知 + 会如何学习 |
| **部分会** | 会什么说什么，猜测时标明假设 |
| **记错** | 面试官纠正时感谢并修正 |

忌：编造 API、坚持错误答案。

## 行为面试：问题解决能力

索引/wiki：`[[问题解决]]`。与开发环境「排错 FAQ」不同，此处指 **STAR 行为题**（冲突、线上事故、技术债抉择等）。

### STAR 模板

| 字母 | 内容 |
| --- | --- |
| **S** | 情境：项目背景、你的角色 |
| **T** | 任务：要达成什么、难点在哪 |
| **A** | 行动：你具体做了什么（用「我」） |
| **R** | 结果：量化或定性成果 + 反思 |

### 示例骨架（Flutter 项目）

```text
S：电商 App 大促前购物车偶发丢单。
T：需在 3 天内定位并保证支付链路稳定。
A：加链路日志 → DevTools 复现 → 发现竞态后改 Repository 串行化 + 补集成测试。
R：丢单率归零，后续需求加 mutex 规范进 Code Review。
```

### 答题要点

1. **选真实经历**：可来自 [[10-项目经验面试题]]、[[21-电商应用实战]] 等。
2. **突出个人贡献**：避免「我们团队」全程无主语。
3. **承认权衡**：没有银弹，说明为何选 A 而非 B。
4. **收尾反思**：若重来会如何改进（监控、测试、架构）。

更多 STAR 结构见 [[09-STAR法则详解]]。

## 反问环节

优质问题示例：

- 团队 Flutter 版本与空安全迁移状态？
- 性能/崩溃的线上监控方案？
- 这个岗位 3～6 个月的成功标准？

避免：只问加班、薪资（终面再谈更合适）。
